58  * # Overview <a name="overview"></a>
59  *
60  * @note This documentation will explain the SWO Driver API's and how to use them.
61  * For full SWOL system documentation including how to setup the parser,
62  * Wireshark dissector, etc, see the <a href="../../Users_Guide.html">User's Guide</a>
63  *
64  * @warning This documentation assumes that the SWOL environment is set up correctly.
65  * This is described in the <a href="../../Users_Guide.html">User's Guide</a>. It is
66  * highly recommended to first perform the Demo from the User's Guide to verify
67  * this before adding SWOL to a separate project.
68  *
69  * The ARM ITM module is used to implement instrumentation and logging using a
70  * low-overhead protocol based on SWIT packets. See the
71  * <a href="https://developer.arm.com/docs/ddi0403/e/armv7-m-architecture-reference-manual">ARM Documentation</a>
72  * for more information regarding the ITM Protocol.
73  * This driver adds another level of parsing (the SWO protocol) above ITM
74  * and abstracts the functionality through the API's documented here.
75  *
76  * The driver also uses sections to place strings symbols in virtual
77  * off-chip memory regions. This reduces the amount of data that needs to be
78  * sent over the physical interface to a (32-bit) pointer pointing to the
79  * symbol in virtual memory. The output ELF file is then parsed to resolve
80  * the symbol based on the received pointer.
81  *
82  * A TI provided tool (Logger) is available for parsing of the ITM data, providing
83  * structured and readable outputs
84  *
85  * # Usage <a name="usage"></a>
86  *
87  * The following includes a section detailing the minimum steps required to add
88  * SWO logging to an embedded project. These common steps will always be required.
89  * After this, examples of each SWO functionality are provided.
90  *
91  * ## Common Setup <a name="setup"></a>
92  *
93  * -# Include the SWO header files:
94  * @code
95  * #include <ti/drivers/SWO.h>
96  * #include <ti/drivers/swo/SWOCC26XX.h>
97  * @endcode
98  *
99  * -# Add the following source files to the project:
100  * - $SWO_PLUGIN_INSTALL$/source/ti/drivers/SWO.c
101  * - $SWO_PLUGIN_INSTALL$/source/ti/drivers/SWO.h
102  * - $SWO_PLUGIN_INSTALL$/source/ti/drivers/swo/SWOCC26XX.c
103  * - $SWO_PLUGIN_INSTALL$/source/ti/drivers/swo/SWOCC26XX.h
104  * - $SWO_PLUGIN_INSTALL$/source/ti/drivers/swo/SWOUtils.h
105  * - $SWO_PLUGIN_INSTALL$/source/ti/boards/CCxxxx_LAUNCHXL/PowerCCXXX_tirtos.c
106  * -# IDE-Specific:
107  * - In CCS, add the SWOL product in Project Properties->General->Project:
108  * @image html add_ccs_product.png
109  * - In IAR, add $SWO_INSTALL_DIR$/source to the Preprocessor Include Directories
110  * in Project->Options->C/C++ Compiler->Preprocessor:
111  * @image html iar_include.png
112  * -# The following steps are optional:
113  * - Configure the pin to be used by the SWO driver by setting the @ref SWO_PIN
114  * preprocessor define. If not defined, this will default to the SWO pin on
115  * the launchpad (IOID_18). Ensure that this pin is not used by other hardware.
116  * - Configure the idle buffer size by setting the @ref SWO_IDLE_BUFFER_SIZE
117  * preprocessor define. See [Deferred Data](#deferred) for more information.
118  * If not defined, this will default to 512.
119  * - Configure the SWO baud rate by setting the @ref SWO_BAUD_RATE preprocessor
120  * define. If not defined, this will default to 12000000.
121  * -# Configure SWO_idleFlush() to be an RTOS Idle task. This is done by adding the
122  * following to the Idle task configuration section of the RTOS configuration
123  * file (xxx.cfg):
124  * @code
125  * Idle.addFunc('&__SWO_idleFlush');
126  * @endcode
127  * -# Update the linker file for the chosen IDE as specified in
128  * [Linker File Modifications](#linker)
129  * -# Open the SWO Driver. See [Opening SWO](#opening)
130  * @code
131  * SWO_Status status;
132  * SWO_open(NULL, &status);
133  * @endcode
134  * -# Enable some logging at compile-time by setting module / level preprocessor defines.
135  * Start by setting ```SWO_LogModule_ENABLE_ALL``` and ```SWO_LogLevel_ENABLE_ALL```. See
136  * [Compile-time Module / Level Enabling](#enabling)
137  *
138  * ## Opening SWO<a name="opening"></a>
139  *
140  * Before logging, the SWO Driver needs to be enabled by a call to SWO_open().
141  * Multiple calls to SWO_open() are supported as only the first call will
142  * configure the ITM module. If multiple successful calls to SWO_open() are
143  * performed, an equal number of calls to SWO_close() need to be performed
144  * before the driver disables logging (stops sending all ITM data). Following
145  * SWO_open(), the user can use the provided logging APIs for instrumentation
146  * of the software, see the following sub sections covering the available
147  * logging APIs and how to use them.
148  *
149  * ## Compile-time Module / Level Enabling<a name="enabling"></a>
150  *
151  * For each API, a @ref SWO_LogLevel and @ref SWO_LogModule need to be provided
152  * as initial arguments. The configuration of modules and levels at
153  * compile-time dictates which SWO API's are compiled into the code. Thus, it is
154  * easy to completely remove all SWO flash consumption by not setting any
155  * SWO_XXX_ENABLE_XXX define's.
156  *
157  * Per default, all log modules and levels are disabled. The user can choose
158  * to enable all of them by defining ```SWO_LogModule_ENABLE_ALL```
159  * and ```SWO_LogLevel_ENABLE_ALL``` in the global project settings. Individual
160  * modules and levels can be enabled by substituting "ALL" with the relevant
161  * module or level. For more information, see the individual module documentation
162  * in @ref SWO_LogModule. For example, a list of defines to enable a subset of
163  * modules and levels could be:
164  *
165  * * ```SWO_LogModule_ENABLE_USER```
166  * * ```SWO_LogModule_ENABLE_DRIVERS```
167  * * ```SWO_LogLevel_ENABLE_ERROR```
168  * * ```SWO_LogLevel_ENABLE_WARNING```
169  *
170  * ## Printf <a name="printf"></a>
171  * Printf-like logging is supported using SWO_printf(). Note that
172  * only 32-bit arguments are supported. SWO_printf_deferred() supports deferring
173  * the arguments (see [Deferred Data](#deferred))
174  *
175  * Here is a sample of various printf calls:
176  *
177  * @code
178  * // Open SWO Driver with default parameters
179  * SWO_Status status;
180  * SWO_open(NULL, &status);
181  *
182  * // Do blocking printf
183  * SWO_printf(SWO_LogModule_User, SWO_LogLevel_INFO, "String without arguments");
184  * SWO_printf(SWO_LogModule_User, SWO_LogLevel_INFO, "String with two argument %d %d", 1, 2);
185  *
186  * // Do deferred printf
187  * SWO_printf_deferred(SWO_LogModule_User, SWO_LogLevel_INFO, "String without arguments");
188  * SWO_printf_deferred(SWO_LogModule_User, SWO_LogLevel_ERROR, "String with one argument %d", 'A');
189  * @endcode
190  *
191  * Here is a sample of what the second printf output above would look like in Wireshark:
192  *
193  * @image html swo_printf.png
194  *
195  * ## LogBuf <a name="logbuf"></a>
196  * SWO_logBuf() can be used to log a raw data buffer of known-size.
197  * The buffer is logged together with a user provided description of
198  * the content. SWO_logBuf_deferred() can be used to defer data (see [Deferred Data](#deferred)).
199  *
200  * Here is a sample of various logBuf calls:
201  *
202  * @code
203  * uint8_t testBuf[6] = {0, 1, 2, 3, 4, 5};
204  *
205  * // Open SWO Driver with default parameters
206  * SWO_Status status;
207  * SWO_open(NULL, &status);
208  *
209  * // Do blocking logBuf
210  * SWO_logBuf(SWO_LogModule_User, SWO_LogLevel_INFO, "This is a test buffer: ", testBuf, 6);
211  *
212  * // Do deferred logBuf
213  * SWO_logBuf_deferred(SWO_LogModule_User, SWO_LogLevel_INFO, "Deferred test buffer ", testBuf, 6);
214  * @endcode
215  *
216  * Here is a sample of what the first log buffer output from above would look like in Wireshark:
217  *
218  * @image html swo_logbuf.png
219  *
220  * ## Events <a name="events"></a>
221  * SWO_event() provides functionality to log a pre-defined event from multiple
222  * places in the code. While printf and logbuf provide
223  * flexibility in the amount of data that can be logged, this comes at the tradeoff of
224  * relatively higher code and processing time overhead. In situations where absolute
225  * minimal invasiveness is desired, SWO_event() should be used.
226  *
227  * @warning An event must be declared prior to using SWO_event() with @ref SWO_CREATE_EVENT
228  *
229  * Here are some samples of various event calls:
230  *
231  * @code
232  * SWO_CREATE_EVENT(SWO_LogModule_User, EV2, "Event 2");
233  *
234  * SWO_event(SWO_LogModule_BLEStack, SWO_LogLevel_INFO, EV0);
235  * SWO_event(SWO_LogModule_User, SWO_LogLevel_INFO, EV1, 1);
236  * SWO_event(SWO_LogModule_User, SWO_LogLevel_INFO, EV2, 1, 2);
237  * SWO_event(SWO_LogModule_User, SWO_LogLevel_WARNING, EV2, 1, 2, 3);
238  * SWO_event(SWO_LogModule_User, SWO_LogLevel_ERROR, EV2, 1, 2, 3, 4);
239  * @endcode
240  *
241  * Here is what the output from the third event call above would look like in Wireshark:
242  *
243  * @image html swo_event.png
244  *
245  * ## Event sets <a name="eventsets"></a>
246  * Event sets can be used to gather multiple logging instructions under one
247  * single event. This can be used by the receiver to easily parse multiple types
248  * of SWO messages. An event set is started by calling SWO_EventSet_init() for a
249  * pre-defined event, which will return a non-zero event set ID if successful.
250  * Records can then be added to the set using SWO_EventSet_printf() and
251  * SWO_EventSet_logBuf(). As in the case for the original printf and
252  * logBuf APIs, there are deferred versions available of each. The event set
253  * must eventually be closed by calling SWO_EventSet_close(). Event sets provide the
254  * highest flexibility in usage at the tradeoff of higher code size and processing
255  * time.
256  *
257  * Here is an example of creating an event set with various records:
258  *
259  * @code
260  * SWO_CREATE_EVENT(SWO_LogModule_User, EV1, "Event 1");
261  *
262  * SWO_EventSet eventSetHandle;
263  * SWO_EventSet_init(SWO_LogModule_User, SWO_LogLevel_INFO, &eventSetHandle, EV1);
264  * SWO_EventSet_printf(SWO_LogModule_User, SWO_LogLevel_INFO, eventSetHandle, "Test printf record %d %d %d", 1, 2, 3);
265  * SWO_EventSet_logBuf(SWO_LogModule_User, SWO_LogLevel_INFO, eventSetHandle, "Test logBuf record: ", testBuf, 6);
266  * SWO_EventSet_printf_deferred(SWO_LogModule_User, SWO_LogLevel_INFO, eventSetHandle, "Deferred test printf record %d %d %d", 1, 2, 3);
267  * SWO_EventSet_logBuf_deferred(SWO_LogModule_User, SWO_LogLevel_INFO, eventSetHandle, "Deferred test logBuf record ", testBuf, 6);
268  * SWO_EventSet_close(SWO_LogModule_User, eventSetHandle);
269  * @endcode
270  *
271  * @image html swo_eventset.png
272  *
273  * ## Deferred Data <a name="deferred"></a>
274  *
275  * Any of the deferred calls above will store the payload data to the idle buffer, the size of which
276  * was configured via @ref SWO_IDLE_BUFFER_SIZE.
277  *
278  * Data will only be transmitted from the idle buffer when SWO_idleFlush() is called. Normally, this
279  * occurs during the Idle task if the RTOS has been configured as stated in [Setup](#setup).
280  *
281  * If at any point a SWO API can not place its payload data in the idle buffer because there is not
282  * enough room, a Buffer Overflow Packet will be sent to the logger. It will appear in the
283  * Wireshark output as such:
284  *
285  * @image html swo_overflow.png
286  *
287  * # Linker File Modifications <a name="linker"></a>
288  * In order to use the SWO Driver, a new ".swo_trace" section needs to be setup in the linker
289  * file. This section is used to store information in the .elf file in order to avoid consuming
290  * flash. The python logger will then pares the .elf file to extract this information.
291  *
292  * The .swo_trace section must be placed in a memory region range that does not overlap
293  * with the RAM or FLASH regions. The section must be placed as "COPY" to ensure the
294  * symbols are available in the ELF output file.
295  *
296  * @warning It is important to ensure that the section is exactly called .swo_trace as this
297  * section name is used by the logger python tool when parsing the .elf file.
298  *
299  * Example CSS linker file modifications:
300  * @code
301  * // System memory map
302  * MEMORY
303  * {
304  * ...
305  * SWO_TRACE (R) : origin = 0x60000000, length = 0xFFFFFFFF
306  * }
307  *
308  * // Section allocation in memory
309  * SECTIONS
310  * {
311  * ...
312  * .swo_trace : > SWO_TRACE, type = COPY
313  * ...
314  * }
315  * @endcode
316  *
317  * Example IAR linker file modifications:
318  * @code
319  * define region TRACES_AREA = mem:[from 0x60000000 to 0x6FFFFFFF];
320  *
321  * ".swo_trace":
322  * place noload in TRACES_AREA { readonly section .swo_trace };
323  * @endcode
324  *
325  * # Power Management <a name="power"></a>
326 
327  * There are several modifications required to the power driver to use SWO with
328  * power management. This is accomplished by compiling the relevant
329  * PowerCCxxxx_tirtos.c file into the project to overwrite the library implemenation
330  * (see [Setup](#setup)). If desired, continue reading this section for more
331  * information on why this is the case.
332  *
333  * The ITM hardware is disabled during CPU low-power modes. To handle this,
334  * SWO_prepareForIdle() needs to be called prior to putting the CPU into low-
335  * power modes. On returning from a low-power mode, SWO_returnFromIdle() need
336  * to be called to re-configure the ITM port and provide time synchronization
337  * with external logger tools. When going into standby, this is handled via the
338  * Power driver notification system. For any other power state such as "idle",
339  * this need to be handled by the code responsible for this mode as no Power
340  * notifications is provided.
341  *
342  * @cond NODOC
343  * ## Tracking Dynamic Memory Allocation ##
344  * Wrapper functions is available to log usage of dynamic memory inside the
345  * application. These wrapper functions replaces the original malloc() and
346  * free() calls with instrumented versions. To use these wrappers, the linker
347  * file need to be contain symbol mapping in accordance with the toolchain used:
348  *
349  * ### TI ARM toolchain linker file ###
350  * @code
351  * --symbol_map=__real_free=free
352  * --symbol_map=malloc=SWO_malloc
353  * --symbol_map=__real_malloc=malloc
354  * --symbol_map=free=SWO_free
355  * @endcode
356  *
357  * ### IAR toolchain ###
358  * TODO
359  * @endcond //NODOC
360  */
